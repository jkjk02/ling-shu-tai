## 目标

1. 为新一轮“后端 API 自动化回归测试”切片建立独立运行台账。
2. 在不引入高风险范围扩张的前提下，把现有手工 HTTP 验证沉淀为可重复执行的自动化回归入口。
3. 固定本轮上游输入、范围边界、验收门槛和文档链，方便后续会话继续扩展测试体系。

## 核心内容

1. 当前项目状态：
   - 最新接管基线为 `docx/07-delivery/20260320-170723-current-product-handoff.md`。
   - 产品已具备 `Skills`、`MCPs`、`Agents`、`Workflows` 的后端 API 与真实手工验证记录。
   - README 已明确“自动化测试体系尚未建立”，这是当前最高优先级未完成项之一。
2. 本轮优先级判断：
   - 上一轮交付已完成，当前不存在阻断缺陷。
   - “继续”推进时，最小且高价值的增量是把既有手工 API 验证沉淀为隔离的自动化回归测试。
   - 由于当前项目无前端自动化测试依赖和浏览器测试基建，本轮优先聚焦后端 API。
3. 本轮锁定范围：
   - 为 FastAPI 后端建立可重复执行的自动化回归测试入口。
   - 覆盖核心资源 API 的成功链路与关键失败链路，至少包括 `Skills`、`MCPs`、`Agents`、`Workflows`、`Dashboard`、`Discovery`。
   - 测试必须运行在隔离数据目录中，不能污染现有 `backend/data` 示例数据。
   - README 与交付文档需要补充测试运行方式和当前覆盖边界。
4. 本轮非范围：
   - 不接入浏览器级 E2E 测试。
   - 不新增 Monaco、外部 MCP 回写或其它产品功能。
   - 不要求在本轮补齐完整 CI 平台接入。
   - 不重构现有业务接口协议，除非测试落地所必需。
5. 验证门槛：
   - 自动化测试命令可在本地重复运行并稳定通过。
   - `npm run build` 继续通过，确保前端未被回归破坏。
   - 如涉及 Python 代码调整，仍需通过最小语法校验或等价验证。
6. 文档约束：
   - 需求、架构、计划、实现、评审、测试、交付各阶段均需形成新的时间戳文档。
   - 下游阶段必须只读取本轮最近一次已批准文档，不得直接依赖聊天结论。

## 风险与待确认项

1. 当前 `backend/requirements.txt` 未声明专门测试框架，架构阶段需要优先选择无需新增外部依赖或能被现有环境承载的方案。
2. 后端配置通过 `get_settings()` 缓存初始化，测试隔离必须显式控制环境变量与缓存清理，否则可能误写真实数据目录。
3. 本轮聚焦 API 回归，不代表浏览器交互、画布拖拽和前端状态流已获得自动化保障。

## 交接输出

1. 允许需求阶段围绕“后端 API 自动化回归测试”建立独立需求基线。
2. 要求架构阶段至少比较两种测试落地路径，并明确隔离数据、可重复执行和维护成本取舍。
3. 要求测试阶段最终用新建的自动化测试入口验证自身，而不是仅重复手工 HTTP 脚本。
