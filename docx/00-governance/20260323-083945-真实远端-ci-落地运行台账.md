## 目标

1. 为“真实远端 CI 落地”建立独立切片台账。
2. 在既有 `verify`、浏览器自动化和 GitHub Actions 骨架基础上，把 CI 补强到可直接上线的最小可交付状态。
3. 保持本轮范围只覆盖远端 CI 稳定性、失败可诊断性和说明文档对齐，不扩展业务功能。

## 风险与待确认项

1. 本轮可交付的是“可直接上线的远端 CI 配置”，不等于本地会话内已经产生远端运行记录。
2. GitHub Actions hosted runner 对 Playwright 依赖、网络和权限更敏感，需要在 workflow 中显式补强。
3. 若不回收 `test-results` 或报告目录，远端失败后排障成本仍然偏高。
4. README 若继续保留“CI 骨架”表述，会造成阶段文档和仓库说明长期失真。
