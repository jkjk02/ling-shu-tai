## 目标

1. 为新一轮“统一验证命令与 CI 接入”切片建立独立运行台账。
2. 在不引入新测试依赖的前提下，把已存在的本地验证能力接入统一入口和仓库内 CI 骨架。
3. 固定本轮范围、验证门槛和交付链，便于后续继续扩展到浏览器自动化。

## 核心内容

1. 当前项目状态：
   - 最新交付基线已具备后端 API 自动化回归和前端构建验证。
   - 当前仍缺少统一的本地质量门禁命令与仓库内 CI 配置。
   - 浏览器级冒烟自动化尚未具备依赖与基建，不适合作为当前最小可落地增量。
2. 本轮优先级判断：
   - 已存在后端自动化回归，但入口仍分散在 README 和文档中。
   - `.github/workflows/` 尚不存在，后续一旦进 Git 托管仓库无法直接复用验证链。
   - 因此本轮锁定为“统一验证命令与 CI 接入”，先把现有验证资产产品化。
3. 本轮范围：
   - 在 `package.json` 提供统一本地验证命令。
   - 增加 GitHub Actions CI 骨架，自动执行 Python 语法校验、后端自动化回归和前端构建。
   - README 和 `docx` 文档同步说明本地与 CI 的运行方式。
4. 本轮非范围：
   - 不新增浏览器自动化框架。
   - 不改造现有业务功能和 API。
   - 不要求本轮接入真实远端仓库或实际触发线上 CI。
5. 验证门槛：
   - 新增统一命令在本地可执行。
   - CI 配置文件语义正确、步骤完整，能表达当前验证链。
   - `npm run build` 与后端自动化回归继续通过。

## 风险与待确认项

1. 当前目录不是 Git 仓库，CI 工作流文件只能作为待接入远端仓库时的骨架，不能在本地真实触发 GitHub Actions。
2. 本地 `package.json` 脚本如果直接绑定项目内 venv，需要在 README 中明确前置条件。
3. 统一命令只能覆盖现有后端回归与构建，不应被误读为“全栈质量门禁已完备”。

## 交接输出

1. 允许需求阶段围绕“统一验证命令 + CI 骨架”建立独立需求基线。
2. 要求架构阶段比较“仅本地脚本统一”与“本地脚本 + GitHub Actions”两种路径，并给出推荐。
3. 要求实施阶段尽量限制写入到 `package.json`、`.github/workflows/`、`README.md`。
