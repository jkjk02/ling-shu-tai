## 目标

1. 为“外部 MCP 回写与安装测试”建立独立切片台账。
2. 在已完成外部 MCP 发现与只读展示的基础上，继续补齐真实可操作闭环。
3. 锁定本轮仅交付可控范围，不把未知外部方言误报为已支持。

## 核心内容

1. 当前产品状态：
   - `cludea` / `opencode` 外部 MCP 已可被发现、查看详情并参与统计。
   - discovered MCP 仍被统一视为只读，尚未支持原文件回写。
   - 自动化回归已覆盖真实 HTTP 回归，但还没有覆盖外部 MCP 的回写闭环。
2. 本轮目标切片：
   - 对“已识别的扁平 JSON 外部 MCP 文件”补齐回写能力。
   - 允许通过 API 和现有 MCP 页面编辑外部 MCP，并把变更写回原始 JSON。
   - 为无真实用户目录样本的情况，继续使用隔离 fixture 做安装式回归测试，不污染正式数据。
3. 本轮范围：
   - `backend/app/services/discovery.py` 增加外部 MCP 可写能力判定、创建、回写、删除与验证辅助。
   - `backend/app/routers/mcps.py` 改为按能力而不是按 `source_kind` 一刀切限制写操作。
   - `src/pages/McpsPage.vue` 保持现有表单结构，只按 `is_writable` 开放 discovered MCP 的编辑删除。
   - `backend/tests/` 扩充外部 MCP 创建、回写、删除和失败分支测试。
4. 本轮非范围：
   - 不承诺兼容未知嵌套方言或带注释 JSON。
   - 不扩展到外部 Skill、Agent、Workflow 回写。
   - 不引入浏览器自动化。
5. 工作包与边界：
   - 工作包 A：后端回写与测试，写入边界 `backend/app/`、`backend/tests/`
   - 工作包 B：前端最小联动与文档更新，写入边界 `src/pages/`、`README.md`、`docx/`
6. 代理协同记录：
   - `requirements_analyst`：提供需求基线草案要点，不写文件。
   - `solution_architect`：提供方案对比与推荐结论，不写文件。

## 风险与待确认项

1. 当前仅基于 flat JSON 样本推进，真实第三方 MCP 方言仍可能需要专用适配器。
2. 回写 discovered MCP 后，Agents 页会读取到更新结果，需避免联动回归。
3. 若环境不存在可用外部目录，本轮创建外部 MCP 的路径只能在测试隔离目录验证。

## 交接输出

1. 需求阶段必须明确“支持矩阵”和“失败语义”，不能只写成功路径。
2. 架构阶段必须比较“直接归一化覆盖写回”与“有限支持矩阵写回”两种方案，并说明为何只支持 flat JSON。
3. 计划阶段必须把测试覆盖扩展到成功与失败两类回归。
