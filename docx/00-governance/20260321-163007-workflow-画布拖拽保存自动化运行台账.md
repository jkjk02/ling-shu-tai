## 目标

1. 为“Workflow 画布拖拽/保存自动化”建立独立执行台账。
2. 在既有 Workflow 真实后端 round-trip 已完成的基线上，补齐浏览器层最小业务闭环。
3. 保持本轮只覆盖拖拽持久化主路径，不扩到视觉回归、异常矩阵和远端 CI。

## 核心内容

1. 当前产品状态：
   - Workflow 页面已有模板加点、连线、拖拽、保存、删除、自动布局和结构校验。
   - 浏览器测试当前只覆盖 Workflows 页面可打开，不覆盖画布交互和保存回读。
   - Playwright 已支持隔离数据目录启动真实后端。
2. 本轮切片范围：
   - 创建唯一命名 Workflow。
   - 新增节点并完成连线。
   - 对节点执行拖拽并形成坐标变化。
   - 保存后刷新回读，并验证位置持久化。
   - 删除清理，确保重复执行不残留测试数据。
3. 本轮非范围：
   - 不做视觉回归。
   - 不做复杂异常矩阵。
   - 不做执行引擎。
   - 不做真实远端 CI。
4. 代理协同记录：
   - `requirements_analyst`：提供需求基线要点。
   - `solution_architect`：提供架构与实施计划要点。
   - `implementation_reviewer`：提供评审结论。
   - `tester`：提供测试结论。
   - 主代理：本地实现、验证、文档归档与集成。
5. 本轮关键实现风险：
   - Workflow 页面 create mode 状态可能与已加载列表选择状态耦合。
   - Workflows 表格刷新时的 loading mask 可能影响自动化点击稳定性。

## 风险与待确认项

1. 画布拖拽受网格吸附、视口和无头浏览器事件时序影响。
2. 若继续依赖纯文案或 Element Plus 内部 DOM，浏览器用例容易脆弱。
3. 当前只覆盖最小成功路径，未覆盖 workflow_invalid 等异常浏览器场景。

## 交接输出

1. 需求阶段必须锁定为 Workflow 浏览器最小闭环，不得扩成完整编排器回归。
2. 架构阶段必须明确采用“最小可测性增强 + 真实 UI 自动化 + 隔离后端”方案。
3. 计划阶段必须把验证收敛到 `npm run test:browser` 和 `npm run verify`。
