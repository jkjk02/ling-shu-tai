## 目标

1. 为“浏览器级冒烟自动化”切片建立独立运行台账。
2. 在已有后端回归和统一验证入口基础上，为前端关键页面补最小浏览器冒烟验证。
3. 固定本轮范围、依赖、验证门槛和交付链，避免把工作扩散成完整 E2E 平台重建。

## 核心内容

1. 当前项目状态：
   - 已具备后端 API 自动化回归、统一验证入口与 GitHub Actions 骨架。
   - 当前前端仍只有手工验证，浏览器层无自动化覆盖。
2. 本轮优先级判断：
   - 当前最明显的自动化缺口已经从后端转移到浏览器层。
   - 由于已完成统一验证入口，本轮适合把前端关键展示与页面可用性补上最小冒烟测试。
3. 本轮范围：
   - 引入浏览器自动化依赖与运行时
   - 覆盖至少 `Dashboard`、`Skills`、`MCPs`、`Agents`、`Workflows` 页面可打开与关键文案/表格可见
   - 尽量把浏览器冒烟接入现有统一验证入口
4. 本轮非范围：
   - 不做完整交互 E2E
   - 不做视觉回归
   - 不重写前端页面结构
5. 验证门槛：
   - 冒烟测试命令可本地执行
   - 既有 `npm run verify` 与后端回归不被破坏
   - README 和 `docx` 同步更新

## 风险与待确认项

1. 浏览器自动化需要新增依赖和浏览器运行时，是当前切片的主要外部前提。
2. 当前仓库没有现成浏览器测试基建，需要架构阶段严格控制为最小冒烟，而不是扩成完整测试平台。

## 交接输出

1. 允许需求阶段围绕“关键页面可打开与关键内容可见”建立需求基线。
2. 要求架构阶段比较最小 Playwright 冒烟方案与更重的完整 E2E 方案，明确推荐最小方案。
