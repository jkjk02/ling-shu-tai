## 目标

1. 把当前仓库内已存在的 GitHub Actions CI 骨架提升为可直接上线的最小可交付方案。
2. 让远端 runner 上的环境准备、验证执行和失败排障具备足够稳定性与可诊断性。
3. 让 README 与 `docx` 对 CI 状态的描述保持一致，不再停留在“仅骨架”表述。

## 核心内容

## 功能模型

1. 远端触发入口
   - 仓库需要支持 `push`、`pull_request` 之外的手动触发入口，便于维护者复跑和验证。
2. 环境准备
   - workflow 需要完成 Node、Python、后端依赖和 Playwright Chromium 依赖准备。
   - 远端 runner 上的浏览器依赖准备应尽量减少环境差异导致的偶发失败。
3. 验证执行
   - 远端 CI 继续以 `npm run verify` 为统一质量闸口，不新增平行命令链。
4. 失败诊断
   - 浏览器自动化失败后，系统需要保留 `test-results` 或浏览器报告等排障产物。
5. 文档对齐
   - README 需要准确说明当前 CI 已具备远端运行配置、覆盖范围与本地仿真方式。

## 行为模型

1. 贡献者推送代码或发起 PR 时：
   - GitHub Actions 自动启动 CI。
   - workflow 按顺序完成 checkout、运行时准备、依赖安装、Playwright 浏览器安装和 `npm run verify`。
2. 维护者需要复跑时：
   - 可通过手动触发入口发起同一条验证链，而不需要改动代码。
3. 任一步骤失败时：
   - 当前 job 必须明确失败。
   - 系统应尽量保留失败产物供后续排查。
4. 全部步骤通过时：
   - 当前 run 以成功结束，作为该提交或 PR 的统一验证结果。

## 数据模型

1. workflow 配置
   - `.github/workflows/ci.yml` 负责定义触发条件、执行环境、步骤顺序和失败产物回收策略。
2. 测试报告产物
   - `test-results/` 和 `playwright-report/` 作为浏览器失败诊断的主要输出目录。
3. 项目说明
   - `README.md` 记录 CI 覆盖范围、本地仿真方式和远端运行前提。
4. 阶段文档
   - `docx/` 用于记录本轮需求边界、实现范围、测试结论和交付限制。

## 范围

1. 本次需求覆盖：
   - GitHub Actions workflow 的远端稳态补强
   - Playwright 报告产物输出
   - README 与阶段文档对齐
2. 本次需求不覆盖：
   - 新增产品功能
   - 新增测试矩阵或多平台 CI
   - 代用户推送代码或触发远端 workflow

## 约束

1. 保持 `npm run verify` 作为唯一主验证入口。
2. 不修改业务功能，不改动前后端资源模型。
3. 变更范围尽量收敛在 `.github/workflows/ci.yml`、`playwright.smoke.config.ts`、`README.md` 和 `docx/`。
4. 本轮只交付“可上线的远端 CI 配置”，不在本地会话内承诺已经完成远端运行。

## 验收标准

1. `.github/workflows/ci.yml` 具备远端可直接使用的触发条件、超时和失败诊断配置。
2. workflow 在远端 runner 上显式准备 Playwright 所需浏览器依赖，而不是仅依赖隐式环境。
3. `npm run verify` 继续作为远端 CI 的主入口。
4. Playwright 在 CI 下能够产出可上传的报告目录或失败产物。
5. README 不再把 CI 描述为“未在线上流水线中实际触发的骨架”，而是准确描述当前可交付边界。
6. 本地至少完成一次 `CI=1 npm run verify` 验证，证明配置没有破坏现有验证链。

## 假设项

1. 目标平台仍是 GitHub Actions。
2. 用户后续会自行将本轮改动推送到远端仓库以触发真实 run。
3. 当前浏览器测试仍使用 Playwright + Chromium 单浏览器模式。
4. GitHub hosted runner 允许使用 `npx playwright install --with-deps chromium`。

## 未决项

1. 失败产物仅在失败时上传，还是成功时也保留报告，由实现阶段在成本和收益间取舍。
2. 是否需要在 workflow 中增加更细的 job 拆分，本轮先交由架构阶段决策。

## 风险与待确认项

1. 本地 `CI=1 npm run verify` 通过，并不自动等于远端 runner 上的所有系统依赖都已满足。
2. Playwright 浏览器安装和浏览器级测试仍是远端环境里最脆弱的环节。
3. 若报告产物未落盘或未上传，远端失败后的问题定位会明显变慢。
4. README 若描述过度乐观，也会造成“已落地”和“待推送验证”之间的边界混淆。

## 交接输出

1. 允许架构阶段比较“维持当前骨架”与“在单 workflow 内补强远端稳态能力”两种路径。
2. 要求计划阶段给出 `.github/workflows/ci.yml`、`playwright.smoke.config.ts`、`README.md` 的明确写入边界。
3. 实现阶段应优先保证稳定性和可诊断性，而不是继续扩展测试覆盖面。
