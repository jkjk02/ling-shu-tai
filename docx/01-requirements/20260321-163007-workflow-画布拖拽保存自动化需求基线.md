## 目标

1. 为 `Workflows` 页面补齐浏览器层最小业务闭环自动化。
2. 范围仅包括：画布节点新增、拖拽、保存、刷新回读、删除清理。
3. 以真实后端持久化结果为准，证明画布交互不只是前端内存态。

## 核心内容

## 功能模型

1. 页面可进入真实 Workflow 编辑态。
2. 自动化可创建独立测试对象。
3. 自动化可在画布中新增节点并完成至少一条连线。
4. 自动化可执行节点拖拽并形成坐标变化。
5. 自动化可保存 Workflow。
6. 自动化可刷新后回读并验证拖拽结果。
7. 自动化可删除并清理测试数据。

## 行为模型

1. 前置条件：
   - 前后端本地服务可用。
   - `Workflows` 页面未退回 mock 数据模式。
2. 主成功路径：
   - 打开 `Workflows` 页面。
   - 新建唯一名称 Workflow。
   - 新增节点。
   - 完成拖拽。
   - 保存 Workflow。
   - 刷新页面并重新定位该 Workflow。
   - 验证节点位置与保存前一致。
   - 删除该 Workflow，并确认清理完成。
3. 失败判定：
   - 页面进入 mock 数据模式。
   - 拖拽后坐标未变化。
   - 保存后未生成可回读记录。
   - 刷新后节点位置回退。
   - 删除后列表仍能看到临时 Workflow。

## 数据模型

1. Workflow 实体关键字段：
   - `name`
   - `nodes`
   - `edges`
   - `viewport`
2. WorkflowNode 本轮必验字段：
   - `label`
   - 位置坐标 `x/y`
   - `config`
3. 测试数据：
   - 使用唯一 Workflow 名称作为浏览器层主要检索键。
   - 临时数据生命周期限定在单次自动化执行内。

## 约束

1. 必须走真实浏览器交互和真实后端提交，不能用 API 替代主路径。
2. 不扩展到视觉回归、复杂异常矩阵、执行引擎、远端 CI。
3. 不要求文件系统级断言。
4. 不把本轮解释为完整 Workflow E2E 回归。

## 验收标准

1. 页面可见 `Workflows 编排`、`Workflow 列表`、`节点模板`、`编排画布`。
2. 自动化能创建唯一命名 Workflow 并在列表中看到它。
3. 自动化能新增节点并通过拖拽使至少一个节点位置发生变化。
4. 点击保存后，Workflow 可被重新加载。
5. 页面刷新后，重新进入该 Workflow 时，被拖拽节点的位置与保存前一致。
6. 删除该 Workflow 后，刷新页面，列表中不再出现该记录。
7. 同一自动化场景重复执行时，不因上次残留数据而失败。

## 风险与待确认项

1. 当前页面若缺少面向画布交互的测试锚点，浏览器自动化稳定性会受影响。
2. 拖拽行为受网格吸附、缩放比例和浏览器事件时序影响。
3. 页面在后端不可用时会退回 mock 数据模式，若未显式拦截会产生假阳性。

## 交接输出

2. 下游实现阶段应以“真实后端、临时数据、拖拽持久化、刷新回读、删除清理”为唯一主路径。
