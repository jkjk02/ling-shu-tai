## 目标

1. 为仓库提供统一的本地验证命令，降低后端回归和前端构建的执行成本。
2. 为未来进入远端代码托管时预置可复用的 CI 工作流骨架。
3. 在不引入新依赖和不改变业务功能的前提下，提高交付链可重复性。

## 核心内容

## 功能模型

1. 本地验证入口
   - 系统需要提供至少一个统一命令，串联后端语法校验、后端自动化回归和前端构建。
   - 统一命令应复用现有能力，而不是引入新的测试框架。
2. CI 工作流骨架
   - 仓库需要提供 `.github/workflows/` 下的 CI 配置文件。
   - 工作流需表达 Node 安装、Python 环境准备、依赖安装、后端校验、前端构建等步骤。
3. 文档说明
   - README 需要明确本地统一命令和 CI 覆盖内容。
   - `docx` 需要记录当前 CI 仍是骨架，不代表已在远端真实运行。

## 行为模型

1. 开发者在本地执行统一命令时：
   - 先执行 Python 语法校验
   - 再执行后端 API 自动化回归
   - 最后执行前端构建
2. 远端仓库触发 CI 时：
   - 自动安装 Node 与 Python
   - 安装前端与后端依赖
   - 执行与本地一致的验证链
3. 失败时：
   - 任一步骤失败都应中止后续步骤并返回非零结果

## 数据模型

1. 脚本配置
   - `package.json` 需增加验证相关脚本键
2. CI 配置
   - `.github/workflows/ci.yml` 需定义触发条件、运行环境与步骤
3. 文档配置
   - README 需说明前置条件：后端 venv、依赖安装、CI 骨架用途

## 约束

1. 不新增浏览器自动化依赖。
2. 不要求当前本地目录真实连接 GitHub。
3. 不重构业务功能和现有测试基座。
4. 本地统一命令允许依赖 `backend/.venv`，但必须在 README 里写清楚。

## 验收标准

1. `package.json` 存在统一验证脚本，可在本地执行通过。
2. `.github/workflows/ci.yml` 存在，且步骤覆盖后端校验、后端回归和前端构建。
3. README 已说明统一命令和 CI 骨架的用途。
4. 原有 `backend/.venv/bin/python -m unittest discover -s backend/tests` 和 `npm run build` 继续通过。

## 假设项

1. GitHub Actions 作为未来最可能接入的远端 CI 目标。
2. 当前仓库继续沿用 Node 和 Python 双运行时。

## 未决项

1. 本地统一入口采用单个 `verify` 脚本，还是拆成 `check:*` + `verify` 组合，由架构阶段定稿。
2. CI 是否额外缓存 pip / npm，由架构阶段在复杂度与收益间取舍。

## 风险与待确认项

1. 当前目录不是 Git 仓库，CI 文件只能作为预置骨架，无法在本地真正触发远端流水线。
2. 若统一脚本过度绑定项目内 venv，会降低跨环境移植性；需通过 README 说明规避误用。

## 交接输出

1. 允许架构阶段比较“仅本地统一脚本”与“本地统一脚本 + GitHub Actions CI”两种路径。
2. 要求计划阶段给出 `package.json`、`.github/workflows/ci.yml`、`README.md` 的明确写边界。
