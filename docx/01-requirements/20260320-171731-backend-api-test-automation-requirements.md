## 目标

1. 为灵枢台 FastAPI 后端建立首个可重复执行的自动化回归入口，替代当前只存在于交付文档中的手工 HTTP 验证。
2. 在不默认新增外网依赖的约束下，覆盖核心资源 API 的成功链路与关键失败链路。
3. 确保测试运行在隔离数据目录和隔离外部发现目录中，不污染现有 `backend/data` 与本机真实 CLI 配置。

## 核心内容

### 功能模型

1. 测试入口需求
   - 提供单一命令入口，可在本地重复执行后端 API 回归测试。
   - 命令必须只依赖当前仓库和已存在的 `backend/.venv` 能力。
2. 覆盖范围需求
   - 至少覆盖 `GET /api/health`
   - 至少覆盖 `Skills` 的列表、创建、冲突或只读失败场景之一
   - 至少覆盖 `MCPs` 的列表、创建、冲突或只读失败场景之一
   - 至少覆盖 `Agents` 的创建、读取或删除闭环
   - 至少覆盖 `Workflows` 的创建、读取、删除，以及 `422 workflow_invalid`
   - 至少覆盖 `Dashboard` 与 `Discovery` 的基础读取
3. 隔离需求
   - 测试运行不得修改仓库内现有 JSON 示例数据
   - 测试发现目录必须使用临时样本，而不是扫描操作者真实家目录
4. 文档需求
   - README 需增加自动化测试运行方式
   - `docx` 文档链需记录覆盖边界、验证方式和已知未覆盖项

### 行为模型

1. 测试启动前：
   - 创建临时数据根目录
   - 创建临时外部 Skill / MCP 发现样本目录
   - 通过环境变量把后端指向临时目录
2. 测试执行中：
   - 启动真实 FastAPI 服务实例
   - 通过 HTTP 请求依次验证健康检查、资源 CRUD、只读保护、工作流校验、统计接口
   - 记录响应码与关键字段断言
3. 测试结束后：
   - 停止测试服务
   - 清理临时目录
   - 不保留脏数据到正式 `backend/data`

### 数据模型

1. 配置数据
   - 需要支持通过环境变量覆盖测试用数据目录
   - 需要支持通过环境变量指定测试用 `codex` / `cludea` / `opencode` 发现根目录
2. 测试样本数据
   - 需要最小 discovered skill 样本
   - 需要最小 discovered mcp 样本
   - 需要 managed 资源创建载荷与 workflow 合法/非法载荷
3. 测试结果数据
   - 以断言为准，不要求额外产出覆盖率报告
   - 若失败，至少能从控制台看出失败接口和错误响应

## 约束

1. 不默认安装新依赖；设计必须在 `httpx` 缺失的前提下仍可落地。
2. 不把本轮范围扩展到前端浏览器自动化测试。
3. 不为了测试改写业务 API 协议。
4. 自动化测试应优先复用现有 FastAPI 应用、路由与发现机制，而不是复制一套假服务。

## 验收标准

1. 存在可执行的后端 API 自动化测试入口，且可稳定通过。
2. 测试运行后，仓库内 `backend/data/*.json` 与 `backend/data/**` 不因测试而被污染。
3. 自动化测试至少能验证一个 discovered skill 只读冲突或读取场景、一个 discovered mcp 读取场景，以及一个 workflow 非法提交 `422 workflow_invalid` 场景。
4. `npm run build` 继续通过。
5. README 已补充自动化测试命令与覆盖说明。

## 假设项

1. 当前 `backend/.venv` 已可运行 `uvicorn` 与现有 FastAPI 应用。
2. 本轮只要求本地回归入口，不要求接入外部 CI 服务。
3. 使用标准库发起 HTTP 请求可以满足本轮“真实 HTTP 回归”目标。

## 未决项

1. 命令入口最终采用直接 `python -m unittest`，还是再包一层脚本，由架构阶段定稿。
2. 测试是否覆盖 `PUT` 更新场景，由实施计划按工作量裁剪。
3. 是否顺带补充更多配置环境变量文档，由实施阶段在不扩 scope 的前提下决定。

## 交接输出

1. 允许架构阶段比较“安装测试依赖 + TestClient”与“标准库 + 真实 HTTP 服务进程”两条路径。
2. 要求架构阶段明确隔离数据目录的配置改动点。
3. 要求计划阶段给出测试目录结构、命令入口和验证顺序。
