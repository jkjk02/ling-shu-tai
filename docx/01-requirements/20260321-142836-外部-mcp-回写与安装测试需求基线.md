## 目标

1. 在不破坏 managed MCP CRUD 和既有 discovered 读链路的前提下，补齐外部 MCP 的最小可回写闭环。
2. 让用户可以对支持的外部 MCP 完成“发现 -> 编辑 -> 回写 -> 再读取验证”的操作。
3. 让测试在隔离目录中覆盖成功与失败两类路径。

## 核心内容

1. 功能模型：
   - FM-1 外部 MCP 识别：继续区分 `managed` 与 `discovered`。
   - FM-2 回写能力判定：对 discovered MCP 明确给出是否可写。
   - FM-3 外部 MCP 编辑：支持的 discovered MCP 与 managed MCP 复用同一表单。
   - FM-4 外部 MCP 回写：保存时必须写回 `source_path`，不能伪装成 managed 副本。
   - FM-5 结果反馈：成功或失败都要返回结构化错误/结果。
2. 行为模型：
   - BM-1 用户浏览 `MCPs` 页面时，可看到 discovered MCP 的可写状态。
   - BM-2 对可写 discovered MCP，允许编辑、保存、删除。
   - BM-3 保存后重新读取同一 MCP 时，应能看到更新后的值和不变的 `source_path`。
   - BM-4 不可写 discovered MCP 继续保持明确拒绝，而不是静默失败。
3. 数据模型：
   - DM-1 延续现有 MCP 统一字段模型。
   - DM-2 对支持写回的外部 JSON，允许把标准字段与 `extra_params` 扁平化写入文件。
   - DM-3 测试必须使用临时 `LINGSHU_*_ROOT` 与临时 `LINGSHU_DATA_DIR`，不得污染仓库默认数据。
4. 验收标准：
   - AC-1 `GET /api/mcps` 可返回至少一个 `is_writable=true` 的 discovered MCP 样本。
   - AC-2 更新该 MCP 后，再次 `GET` 能看到新值已生效且 `source_path` 未变。
   - AC-3 删除该 MCP 后，再次列表中不再出现。
   - AC-4 `POST /api/mcps` 允许在支持目录下创建 `source_kind=discovered` 的外部 MCP。
   - AC-5 不支持写回或不可写路径返回结构化 `403`/`409` 错误，而不是通用 `500`。
   - AC-6 现有 managed MCP CRUD 与 discovered 发现统计不回归。

## 风险与待确认项

1. 当前需求只承诺支持扁平 JSON MCP 文件，不承诺保真回写未知嵌套结构。
2. 安装测试在本轮定义为“隔离目录中的发现/回写/再发现回归”，不等同于真实 CLI 进程调用。
3. 若后续拿到真实样本，应再补一轮适配器扩展而不是放宽本轮承诺。

## 交接输出

1. 允许架构阶段采用“有限支持矩阵 + 文件回写服务”的实现路径。
2. 允许计划阶段将前端改动限制为已存在按钮按 `is_writable` 自动开放，不额外新增复杂模式切换。
