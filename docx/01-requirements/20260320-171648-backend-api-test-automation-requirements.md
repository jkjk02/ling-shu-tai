## 目标

1. 为灵枢台 FastAPI 后端建立可重复执行的自动化回归测试基线，替代当前仅依赖手工 HTTP 验证的状态。
2. 让后续开发在修改 `Skills`、`MCPs`、`Agents`、`Workflows`、`Dashboard`、`Discovery` 相关接口时，能快速发现行为回归。
3. 确保测试运行不会污染现有 `backend/data` 示例数据，并能被 README 清晰说明。

## 核心内容

### 范围与边界

1. 本轮范围：
   - 建立后端 API 自动化测试入口。
   - 覆盖核心资源的成功链路、关键失败链路和聚合接口响应。
   - 补充测试运行说明与当前覆盖声明。
2. 本轮非范围：
   - 浏览器级 E2E、画布交互自动化、前端组件测试。
   - CI 平台接入、覆盖率平台、质量门禁平台化。
   - 产品新功能开发和接口协议重设计。
3. 约束条件：
   - 测试必须基于当前代码库可获得的本地环境执行。
   - 测试结果必须可重复，不能依赖人工预置真实数据。
   - 测试不应破坏已有手工启动方式与现有数据目录内容。

### 功能模型

1. 测试执行者：
   - 开发者在本地执行单一命令或明确入口，即可跑完后端 API 回归测试。
2. 被验证对象：
   - `GET /api/health`
   - `Skills` 资源的列表、创建、读取、更新、删除，以及只读冲突或锁定场景
   - `MCPs` 资源的列表、创建、读取、更新、删除，以及只读冲突或锁定场景
   - `Agents` 资源的列表、创建、更新、删除
   - `Workflows` 资源的创建、读取、更新、删除，以及结构校验失败场景
   - `GET /api/dashboard`
   - `GET /api/discovery`
3. 产出物：
   - 自动化测试代码
   - 可执行测试命令
   - README 级别的运行说明

### 行为模型

1. 正常回归场景：
   - 测试运行时自动准备隔离环境与初始数据。
   - 测试按资源边界验证 API 响应码、核心字段和关键聚合结果。
   - 测试结束后不保留会污染仓内示例数据的副作用。
2. 失败识别场景：
   - 当接口返回错误状态码、错误码结构或关键字段不符合预期时，测试应直接失败。
   - 当 workflow 提交引用不存在 Agent、出现非法连线或重复 ID 时，测试应验证后端返回明确失败结果。
   - 当发现只读 discovered 资源不可写时，测试应验证服务端拒绝写操作。
3. 维护演进场景：
   - 后续新增接口时，现有测试组织方式应允许增量补充，而不需要推倒重来。
   - 当示例数据或发现目录变化时，测试应保持边界清晰，能明确区分“测试隔离数据”和“项目默认数据”。

### 数据模型

1. 测试输入数据：
   - 隔离数据目录下的 `skills.json`、`mcps.json`、`agents.json`、`workflows.json`
   - 可控的外部发现目录样本，用于 discovered skill / MCP 的只读验证与聚合统计验证
2. 测试输出数据：
   - 用例执行结果
   - 关键断言结果，包括响应码、错误码、聚合计数、核心字段回读值
3. 数据约束：
   - 测试样本必须最小可读，避免和生产示例数据耦合。
   - 隔离数据应覆盖 managed 与 discovered 两类来源。
   - 工作流样本至少包含节点、连线、viewport，以及合法与非法两类 payload。

### 非功能需求

1. 可重复性：同一环境下重复执行应得到一致结果。
2. 隔离性：测试不得修改 `backend/data` 现有 JSON 文件内容。
3. 可维护性：测试结构应按资源或能力边界组织，便于后续增量扩展。
4. 可读性：失败输出应足以定位是哪个接口、哪类断言失败。
5. 轻量性：优先复用现有环境能力，不把本轮变成大型测试基建建设项目。

### 验收标准

1. 存在明确、可执行的后端 API 自动化测试入口，并记录在 README 中。
2. 自动化测试至少覆盖以下能力：
   - `health`
   - `Skills` managed CRUD 与只读冲突/锁定
   - `MCPs` managed CRUD 与只读冲突/锁定
   - `Agents` CRUD
   - `Workflows` CRUD 与 `workflow_invalid` 失败路径
   - `Dashboard` 聚合接口
   - `Discovery` 状态接口
3. 测试运行后，仓内现有 `backend/data` 文件保持不变。
4. 至少有一组断言验证 discovered 资源能被读取或统计，但不能被写入。
5. 本轮交付文档中明确当前自动化测试的覆盖边界与未覆盖部分。

### 假设与边界

1. 当前本地 Python 环境已能运行后端服务依赖。
2. 现有 FastAPI 应用工厂和配置层允许通过环境或依赖注入实现测试隔离。
3. 本轮用户接受“先补后端 API 回归、暂不补前端自动化”的分阶段推进策略。

## 风险与待确认项

1. 若当前依赖环境不包含专门测试框架，则需要在架构阶段决定采用内建能力还是引入最小新依赖。
2. `DiscoveryService` 依赖外部目录扫描，需求上要求可测，但样本目录组织方式仍待架构阶段细化。
3. 若现有代码对路径和缓存的耦合高于预期，可能需要最小重构以满足隔离性需求。
4. 本轮不覆盖前端与浏览器交互自动化，因此仍存在 UI 级回归盲区。

## 交接输出

1. 下游架构阶段必须基于本文件设计测试组织方式、隔离策略和执行入口。
2. 下游计划阶段必须把实现任务按“测试基础设施 / 资源回归用例 / 文档与验证”这类边界拆分，且写集不重叠。
3. 下游测试阶段必须用本轮新建的自动化测试入口完成自验证，而不是重复手工 HTTP 验证。
4. 本文件建议标题：`后端 API 自动化回归测试需求基线`
5. 本文件建议 slug：`backend-api-test-automation-requirements`
6. 本文件为后续架构、开发、测试的唯一需求输入。
