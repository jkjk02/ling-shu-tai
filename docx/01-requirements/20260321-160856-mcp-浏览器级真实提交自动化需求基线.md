## 目标

1. 在现有浏览器冒烟基础上，补齐 `managed MCP` 的浏览器级真实创建提交、结果校验与删除清理。
2. 让前端页面对真实 `POST /mcps` 与 `DELETE /mcps/{id}` 的联通性得到自动化验证。
3. 保持本轮范围最小，只覆盖“新建并保存成功、界面结果正确、测试后无残留数据”。

## 核心内容

## 功能模型

1. 真实创建提交
   - 从 `MCPs` 页面进入“新建 MCP”对话框。
   - 使用有效表单数据创建一个 `source_kind=managed` 的 MCP。
   - 触发真实保存，而不是仅校验弹窗可见性。
2. 结果校验
   - 保存成功后，页面应反映新建结果。
   - 新建记录应在列表或详情区域可见，且关键字段与输入一致。
   - 新建记录应表现为可写的 `managed` MCP。
3. 清理回收
   - 测试结束前删除本次创建的 MCP。
   - 删除成功后，页面中不应残留本次测试数据。

## 行为模型

1. 前置阶段
   - 启动真实前端与真实后端服务。
   - 进入 `MCPs` 页面，并确认当前不是 mock 回退状态。
2. 创建阶段
   - 打开“新建 MCP”对话框。
   - 保持写入目标为 `managed`。
   - 填写最小有效字段并提交保存。
3. 校验阶段
   - 等待保存完成后的页面稳定状态。
   - 校验列表展示、详情展示和来源类型。
4. 清理阶段
   - 对刚创建的记录执行删除。
   - 确认删除反馈与列表移除结果。

## 数据模型

1. 测试输入数据
   - 名称：本轮唯一，避免与既有数据冲突。
   - 所属 CLI 工具：使用页面允许的有效值。
   - 模型名称：有效非空值。
   - 扩展参数：合法 JSON 对象。
   - 写入目标：固定为 `managed`。
2. 预期输出数据
   - 新建 MCP 的名称、模型名称、来源类型、可写状态。
   - 删除后的目标记录不存在状态。

## 约束

1. 必须走真实浏览器交互和真实后端提交，不能用 mock 提交替代。
2. 仅覆盖 `managed MCP`，不得扩展到 `discovered` 写回。
3. 测试必须自清理，不能污染默认数据集。
4. 不以新增复杂测试框架或重构页面为前提。

## 验收标准

1. 浏览器测试中存在独立场景，覆盖 `managed MCP` 的“创建提交 -> 结果校验 -> 删除清理”闭环。
2. 该场景运行时，页面真实触发保存，且新建结果在界面可见。
3. 校验结果至少包含：
   - 名称正确。
   - 模型名称正确。
   - 来源类型为 `managed`。
   - 记录可写。
4. 测试结束后，创建的数据被删除，重复执行不产生残留影响。
5. 新增场景后，既有页面冒烟能力仍保持可运行。

## 风险与待确认项

1. 若测试数据命名不唯一，可能与既有 MCP 发生冲突，导致非业务性失败。
2. 若删除清理失败，可能污染后续运行环境并放大串行失败。
3. 当前仅验证最小成功链路，尚未覆盖保存失败、冲突、只读拒绝等异常场景。

## 交接输出

1. 为架构与计划阶段提供单一明确范围：只实现 `managed MCP` 浏览器级真实提交最小闭环。
2. 明确下游需围绕“真实提交、页面校验、自清理”三件事展开，不得扩展需求范围。
