## 目标

1. 把现有 GitHub Actions CI 从最小骨架补强到可直接上线、可重复运行、失败后可诊断的版本。
2. 保持本地与远端继续复用同一条主验证链 `npm run verify`。
3. 同步 README 和阶段文档，去掉“仅骨架”的过时描述。

## 核心内容

1. 实现概览：
   - 更新 [ci.yml](/home/lst/ling-shu-tai/.github/workflows/ci.yml)，新增 `workflow_dispatch`、最小只读 `permissions`、concurrency、`timeout-minutes`、`CI=true`、pip cache、升级 pip、`npx playwright install --with-deps chromium` 和 artifact 上传。
   - 更新 [playwright.smoke.config.ts](/home/lst/ling-shu-tai/playwright.smoke.config.ts)，新增 `isCI`、CI 下 HTML/JUnit reporter、稳定 `outputDir`、`retries`、失败截图和重试 trace。
   - 更新 [README.md](/home/lst/ling-shu-tai/README.md)，将“CI 骨架”改为“GitHub Actions CI”，补充远端运行配置和 `CI=1 npm run verify` 本地仿真说明。
2. 关键实现策略：
   - 远端 workflow 继续只保留单个 `verify` job，不引入多 job 和矩阵复杂度。
   - 验证入口继续收敛在 `npm run verify`，避免在 YAML 内再造一套业务验证逻辑。
   - 失败诊断优先保留 `test-results/`、`playwright-report/`，而不是依赖控制台长日志。
   - Playwright 在 CI 下开启 `retries: 1` 与 `trace: on-first-retry`，让重试 trace 真正落盘。
3. 缺陷闭环：
   - 首次增强时尝试在 CI 下保留视频，但本地 `CI=1` 仿真发现缺少 `ffmpeg` 会导致 `browserContext.newPage` 直接失败。
   - 已将 `video` 调整为 `off`，保留 `trace + screenshot + HTML report` 作为诊断主手段，既减少环境依赖，也保持失败可排查。
4. 代码归属：
   - `worker-1`：`.github/workflows/ci.yml`、`playwright.smoke.config.ts`
   - 主代理：`README.md`、`docx/`、验证与集成
5. 实现后验证：
   - `CI=1 npm run verify` 通过
   - `test-results/playwright/results.xml` 已生成
   - `playwright-report/index.html` 已生成

## 风险与待确认项

1. 本轮交付的是“可直接上线的远端 CI 配置”，不等于本会话已经替用户触发了 GitHub 远端 run。
2. GitHub hosted runner 基础镜像若发生变化，Playwright 或系统依赖仍可能需要微调。
3. 当前 artifact 上传采用 `if: always()`，成功与失败都会保留报告目录，会增加一定存储占用。
