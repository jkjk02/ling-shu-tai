## 实施顺序

1. 先修改 `.github/workflows/ci.yml` 与 `playwright.smoke.config.ts`。
2. 再更新 `README.md`，确保仓库说明与实现一致。
3. 最后执行 `CI=1 npm run verify`，并把结果写入实现记录、评审、测试和交付文档。

## 验证计划

1. `npm run check:backend`
2. `npm run test:backend`
3. `npm run build`
4. `npm run test:browser`
5. `CI=1 npm run verify`

## 完成定义

1. workflow 文件具备可直接推送到 GitHub 使用的远端稳态配置。
2. Playwright 在 CI 下会输出可上传的失败排障产物。
3. README 已准确说明当前 CI 状态、本地仿真方式和远端触发前提。
4. `CI=1 npm run verify` 在本地继续通过。

## 风险与待确认项

1. 如果 worker 修改了报告目录名，README 和测试文档必须同步更新，避免产物路径不一致。
2. 本轮不直接推送远端，因此“真实 run 已产生”不属于本地完成定义。
3. 若 hosted runner 后续发生基础镜像变化，仍可能需要追加微调。
