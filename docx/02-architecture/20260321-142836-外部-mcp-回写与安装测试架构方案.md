## 目标

1. 为外部 MCP 回写闭环选择可控实现方案。
2. 把未知方言风险限制在当前样本支持矩阵内。

## 核心内容

1. 方案对比：
   - 方案 A：对所有 discovered MCP 直接按归一化模型覆盖原文件。
     - 优点：改动少。
     - 缺点：容易破坏未知 JSON 结构，风险过高。
   - 方案 B：只对“已识别的 flat JSON MCP 文件”开放回写，并由专门的文件服务负责创建/更新/删除。
     - 优点：范围清晰，测试可控，可逐步扩展。
     - 缺点：支持矩阵有限。
2. 推荐方案：方案 B。
3. 模块边界：
   - `DiscoveryService`：继续负责扫描、解析、合并，并新增对 flat JSON 文件的写入能力与候选目录解析。
   - `mcps.py`：创建、更新、删除时按“是否 discovered 且可写”分流到外部文件写回逻辑，否则仍走 managed 资源服务。
   - `McpsPage.vue`：不增加新模式，只复用现有 `isWritable` 联动按钮。
   - `backend/tests/support.py`：继续用临时目录起真实服务进程。
4. 写回策略：
   - 外部 MCP 文件采用标准字段 + `extra_params` 顶层展开写回。
   - `id` 由文件相对路径派生，更新名称不改变 `id`，删除按 `source_path` 删除原文件。
   - 新建外部 MCP 时，落到候选根目录下的 `mcps/<slug>.json`。
5. 质量属性取舍：
   - 优先安全性和可验证性，不优先覆盖全部外部方言。
   - 优先复用当前 API 和 UI，不新增独立安装向导。

## 风险与待确认项

1. 真实目录中文件权限或软链接可能导致写回失败，本轮只做常规文件校验。
2. 通过 `extra_params` 顶层展开能兼容当前样本，但不保证覆盖后续复杂结构。

## 交接输出

1. 允许计划阶段把实现拆成“后端写回与测试”和“前端最小联动与文档”两块。
2. 不要求新增额外后端服务类，允许在 `DiscoveryService` 内集中实现本轮能力。
