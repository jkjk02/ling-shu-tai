## 目标

1. 在现有 Workflow 页面和 Playwright 真前后端联调基础上，补齐画布拖拽/保存自动化。
2. 严格控制范围，只覆盖真实浏览器交互与持久化回读，不扩展到视觉回归和远端 CI。

## 核心内容

1. 方案对比：
   - 方案 A：纯黑盒 UI 自动化。
     - 优点：不改产品页面结构。
     - 缺点：依赖文案、类名和 Element Plus 结构，稳定性弱。
   - 方案 B：最小可测性增强 + UI 自动化。
     - 优点：只补最小测试锚点，不改协议，稳定性更高。
     - 缺点：需要少量页面模板改动。
2. 推荐方案：方案 B。
3. 模块边界：
   - `WorkflowsPage.vue`：增加稳定定位点，并修正 create mode 状态逻辑。
   - `tests/browser/workflow-canvas.spec.ts`：新增最小 Workflow 浏览器闭环用例。
   - `playwright.smoke.config.ts` / `playwright_backend.py`：继续复用既有隔离后端能力。
   - `README.md`：同步 Workflow 浏览器覆盖边界。
4. 关键技术策略：
   - 真实浏览器交互仍通过节点模板按钮、端口按钮、鼠标拖拽和保存按钮完成。
   - 节点位置回读通过节点内联 `left/top` 渲染结果校验，不引入新协议。
   - 刷新后显式按唯一 Workflow 名称重新定位，不依赖默认选中态。

## 风险与待确认项

1. 拖拽动作在无头浏览器里受视口和网格吸附影响。
2. 表格刷新 loading mask 可能影响自动化点击稳定性。
3. 当前仍未覆盖 workflow_invalid 等异常浏览器场景。

## 交接输出

1. 架构建议按“最小可测性增强 + 真实 UI 自动化”落地。
2. 实施阶段应严格围绕主路径，不扩展异常矩阵和视觉回归。
