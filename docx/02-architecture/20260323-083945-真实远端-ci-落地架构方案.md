## 目标

1. 在不扩展产品功能的前提下，为仓库选择最合适的远端 CI 补强路径。
2. 保证远端 runner 上的验证流程尽量复用现有 `npm run verify`，同时补足可观测性。

## 核心内容

## 方案比较

### 方案 A：维持现有单 job 骨架，只保留最小步骤

1. 做法
   - 保留现有 `checkout -> install -> npm run verify` 流程，不额外增加手动触发、超时、并发控制和失败产物处理。
2. 优点
   - 改动最少。
   - 几乎不需要同步调整 Playwright 配置。
3. 缺点
   - 远端 runner 上的浏览器依赖仍偏脆弱。
   - 失败后几乎没有附加诊断产物。
   - README 仍难以支撑“真实远端可运维”的目标。

### 方案 B：继续保留单 job，但补强远端稳态和失败诊断能力

1. 做法
   - 在现有 workflow 上增加 `workflow_dispatch`、`permissions`、`timeout-minutes`、concurrency。
   - 使用 `actions/setup-python` pip cache。
   - 将 Playwright 安装改为 `npx playwright install --with-deps chromium`。
   - 让 Playwright 在 CI 下输出 HTML 报告或稳定的失败产物目录。
   - 在 workflow 失败时上传 `test-results/`、`playwright-report/`。
2. 优点
   - 保持单条主验证链，复杂度低。
   - 直接解决当前最关键的远端稳定性与失败排障问题。
   - 代码改动面小，适合本轮“最小落地”目标。
3. 缺点
   - 仍然只有单 job，不能提供更细粒度的并行或按阶段门禁。
   - 成功时默认不一定保留完整报告，观测能力仍以失败场景优先。

### 方案 C：拆成多 job 或矩阵式 CI

1. 做法
   - 将 backend、build、browser tests 拆为多个 job，或引入 Node/Python/OS 矩阵。
2. 优点
   - 可获得更细的阶段可视化和更强扩展性。
3. 缺点
   - 对当前仓库属于过度设计。
   - 增加维护成本和失败排障路径复杂度。
   - 与本轮“最小可交付”目标不匹配。

## 推荐方案

1. 采用方案 B。
2. 理由：
   - 当前 `CI=1 npm run verify` 已能在本地通过，说明主验证链已经足够稳定，优先补齐远端 workflow 形态即可。
   - 方案 B 在改动范围最小的前提下，正好覆盖当前缺失的触发、稳态和诊断能力。
   - 相比多 job 或矩阵式方案，方案 B 更符合本轮交付节奏。

## 目标架构

1. `.github/workflows/ci.yml`
   - 维持单 job `verify`。
   - 增加 `workflow_dispatch`、`permissions: contents: read`、`timeout-minutes`、concurrency。
   - 使用 Node 22 与 Python 3.12。
   - 打开 pip cache，并在失败时上传 Playwright 相关产物。
2. `playwright.smoke.config.ts`
   - 在 CI 下输出稳定的报告目录，供 workflow 上传。
   - 本地保持现有使用方式，不引入额外人工步骤。
3. `README.md`
   - 更新为“GitHub Actions CI 已具备远端可运行配置”。
   - 写清楚本地仿真验证方式和远端触发前提。

## 质量属性与取舍

1. 一致性
   - 继续复用 `npm run verify` 作为本地和远端统一入口。
2. 稳定性
   - 倾向显式安装 Playwright 系统依赖，而不是依赖 runner 偶然预装。
3. 可诊断性
   - 优先保障失败后能拿到 `test-results`、`playwright-report`。
4. 低复杂度
   - 不做多 job 或矩阵拆分，避免当前切片膨胀。

## 风险与待确认项

1. GitHub hosted runner 的具体系统环境仍可能变化，本轮通过 `--with-deps` 和报告上传降低风险，但不能完全消除外部平台波动。
2. 若 Playwright HTML 报告输出目录配置不当，artifact 上传可能拿不到有效内容。
3. 本轮不代替用户执行推送，因此远端 run 仍需在后续由用户触发确认。
