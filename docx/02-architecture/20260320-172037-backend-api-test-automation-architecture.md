## 核心内容

## 方案比较

### 方案 A：新增测试依赖，使用 `pytest` / `httpx` / `fastapi.testclient`

1. 做法
   - 安装 `pytest` 与 `httpx`
   - 直接在进程内创建 FastAPI app，使用 `TestClient` 执行接口回归
2. 优点
   - 测试编写体验更直接，端口和子进程管理更简单
   - 断言能力成熟，未来扩展 fixture 更方便
3. 缺点
   - 当前环境缺少 `httpx`，引入新依赖需要额外安装链路
   - 对“继续推进”这一轮来说，新增依赖与环境漂移风险高
   - 属于应用内调用，不如真实 HTTP 子进程方案贴近现有手工验证路径

### 方案 B：标准库 `unittest` + `subprocess` 启动 `uvicorn` + `urllib` 发起真实 HTTP

1. 做法
   - 使用 Python 标准库构建测试基座
   - 测试阶段动态创建临时数据目录和 discovery 样本目录
   - 通过环境变量指向临时目录，子进程启动真实 `uvicorn`
   - 使用 `urllib.request` 访问真实 HTTP 接口并断言响应
2. 优点
   - 不依赖外网安装新包，适配当前仓库约束
   - 与既有手工验证方式最接近，验证保真度更高
   - 可直接证明配置隔离是否生效
3. 缺点
   - 需要自行处理服务启动、端口轮询与子进程清理
   - 断言和请求封装比专用测试框架略繁琐

## 推荐方案

1. 采用方案 B。
2. 主要理由：
   - 当前约束下，依赖最小化优先级高于测试编写舒适度。
   - 本轮目标是把“真实 HTTP 手工验证”沉淀为自动化回归，而不是建设完整测试平台。
   - 通过真实 `uvicorn` 子进程可以同时验证配置覆盖、数据隔离和路由装配，不会因为进程内 shortcut 掩盖问题。

## 目标架构

1. 配置层
   - 在 `backend/app/config.py` 新增数据目录环境变量覆盖能力，例如 `LINGSHU_DATA_DIR`。
   - 保持 discovery 根目录继续由现有环境变量控制。
2. 测试层
   - 新增 `backend/tests/test_api_regression.py`。
   - 测试基座负责：
     - 创建临时数据目录
     - 写入最小 discovered skill / MCP 样本目录
     - 选择可用本地端口
     - 启动与停止 `uvicorn` 子进程
     - 统一封装 GET / POST / DELETE 请求与 JSON 断言
3. 文档层
   - README 增加自动化测试命令、覆盖范围和限制说明。

## 模块边界

1. `backend/app/config.py`
   - 仅负责让测试可切换数据目录，不承担测试逻辑。
2. `backend/tests/test_api_regression.py`
   - 负责测试样本准备、服务生命周期管理和 API 断言。
3. `README.md`
   - 负责暴露用户运行方式与当前覆盖边界。

## 质量属性与取舍

1. 隔离性：优先级最高。测试不能污染仓库内正式 JSON 数据。
2. 真实性：优先级高。选择真实 HTTP 访问而不是伪造 service 调用。
3. 依赖控制：优先级高。避免本轮因安装缺失依赖而卡住。
4. 可维护性：通过少量测试助手函数控制复杂度，接受比 `pytest` 略重的样板代码。

## 风险与待确认项

1. `uvicorn` 子进程若启动失败，测试需要输出足够日志，否则定位成本高。
2. 若未来接入 CI，当前方案可能还需要再包一层脚本或拆出共享助手。
3. `LINGSHU_DATA_DIR` 作为新环境变量，需要在 README 中说明其用途，避免误解。

## 交接输出

1. 允许计划阶段按“配置隔离 + 测试基座 + README”组织实施。
2. 要求实现阶段只做最小必要改动，不把测试基座反向耦合进业务代码。
3. 要求测试阶段使用新增自动化命令完成自验证，并继续跑 `npm run build`。
