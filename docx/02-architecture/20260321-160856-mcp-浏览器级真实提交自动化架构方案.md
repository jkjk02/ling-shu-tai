## 目标

1. 在现有 Playwright 真前后端联调基础上，补齐 `managed MCP` 的浏览器级真实创建提交自动化。
2. 验证链必须覆盖真实表单填写、真实提交、真实后端写入、测试后自清理或隔离。
3. 严格控制范围：本轮不扩到 external/discovered MCP、新增复杂编辑流、远端 CI。

## 核心内容

1. 方案对比：
   - 方案 A：共享数据目录 + 用例内自清理。
     - 优点：改动最少，接入最快。
     - 缺点：一旦删除失败会污染真实数据；重复跑易产生脏状态。
   - 方案 B：隔离数据目录 + 用例内删除双保险。
     - 优点：真实度足够，污染风险最低，失败也不影响默认数据。
     - 缺点：需要补一层测试运行时隔离脚本。
2. 推荐方案：方案 B。
3. 模块边界：
   - `backend/tests/playwright_backend.py`：为浏览器测试启动隔离数据目录下的真实后端。
   - `playwright.smoke.config.ts`：把浏览器测试后端入口切到隔离脚本。
   - `tests/browser/mcp-smoke.spec.ts`：增加 managed MCP 真实创建、刷新校验、详情校验、删除清理用例。
   - `README.md`：同步运行方式、隔离策略和覆盖边界。
4. 运行策略：
   - 继续使用真实 `uvicorn` + Vite 开发服务。
   - 后端通过 `LINGSHU_DATA_DIR` 指向临时数据副本。
   - 保持单 worker 串行执行，避免并发测试相互干扰。
5. 断言策略：
   - 不依赖 toast 作为唯一成功条件。
   - 必须断言创建后列表可见、刷新后仍可见、详情区可见、删除后移除。

## 风险与待确认项

1. 隔离配置失效时可能误写默认 `backend/data`。
2. Element Plus 弹窗和异步列表刷新若选择器不稳，浏览器断言可能抖动。
3. 当前架构只覆盖成功路径，不覆盖异常矩阵。

## 交接输出

1. 允许计划阶段按“隔离运行时 / 浏览器用例 / README 与验证”三块直接落地。
2. 评审重点：
   - 是否仍是最小浏览器切片。
   - 是否真的避免污染默认数据。
   - 是否没有扩展到 discovered/external 链路。
