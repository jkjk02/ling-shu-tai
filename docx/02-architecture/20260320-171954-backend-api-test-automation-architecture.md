## 目标

1. 为后端 API 自动化回归测试选择可落地、可隔离、可维护的实现路径。
2. 在当前环境约束下，明确测试执行入口、隔离策略、样本组织方式和必要的最小代码改造边界。

## 核心内容

### 候选方案比较

1. 路径 A：新增测试依赖，采用 `pytest` / `httpx` / `fastapi.testclient`
   - 优点：
     - 用例写法简洁，断言与 fixture 生态成熟。
     - 无需启动独立进程即可直接驱动 ASGI 应用。
   - 缺点：
     - 当前 `backend/.venv` 缺少 `httpx`，`fastapi.testclient` 现状不可用。
     - 本轮目标是快速补齐回归基线，引入新依赖会增加环境准备和文档成本。
     - 当前无 Git/CI 基线，先扩依赖再建体系性价比较低。
2. 路径 B：使用 Python 标准库 `unittest` + 子进程 `uvicorn` + `urllib.request`
   - 优点：
     - 完全基于现有虚拟环境可用能力，无需新增依赖。
     - 通过真实 HTTP 回路验证，更贴近当前手工验证方式。
     - 易于在测试过程中控制独立端口、独立数据目录和独立发现目录。
   - 缺点：
     - 用例辅助代码比 `TestClient` 路线更多。
     - 进程启动与端口管理需要额外封装，测试速度略慢。
3. 推荐结论：采用路径 B。
   - 推荐理由：
     - 满足“轻量落地、隔离运行、可重复执行”的核心需求。
     - 规避当前环境中 `httpx` 缺失的直接阻塞。
     - 与既有真实 HTTP 验证链路一致，能把手工验证平滑迁移为自动化回归。

### 模块边界

1. `backend/tests/`
   - 存放测试入口、HTTP 辅助工具、隔离环境搭建逻辑和 API 回归用例。
2. `backend/tests/fixtures/`
   - 存放 discovered skill / MCP 的最小样本目录结构与测试数据。
3. `backend/app/config.py`
   - 增加或调整对测试隔离路径的读取能力，使应用可在测试时指向临时数据目录。
4. `README.md`
   - 记录自动化测试命令、前置条件、覆盖边界。

### 关键设计

1. 隔离数据策略：
   - 测试启动前在临时目录创建独立 `data/` 树。
   - 应用通过环境变量或等价可注入配置读取该目录，而不是写入默认 `backend/data`。
   - 每轮测试结束后自动清理临时目录。
2. 发现目录策略：
   - 为 `codex`、`cludea`、`opencode` 创建最小外部样本目录。
   - 用环境变量把 `DiscoveryService` 的候选根目录指向测试样本，确保 discovered 资源可稳定复现。
3. 进程与请求策略：
   - 由测试辅助模块选择空闲本地端口。
   - 子进程启动 `uvicorn app.main:app --app-dir backend` 或等价入口。
   - 测试通过 `urllib.request` 发送真实 HTTP 请求，并统一解析 JSON 响应。
4. 用例组织策略：
   - 按资源能力拆分测试类或测试文件：基础健康检查、资源 CRUD、只读保护、workflow 校验、聚合接口。
   - 保留少量端到端级断言，不追求穷举全部字段。

### 质量属性取舍

1. 可维护性优先于极致速度：
   - 接受子进程 HTTP 测试稍慢，以换取无需额外依赖和更贴近真实行为。
2. 隔离性优先于零改动：
   - 允许对配置层进行最小修改，只要目标是消除对真实数据目录的写入风险。
3. 可扩展性保持最低必要：
   - 本轮只铺设后端 API 回归结构，未来若引入 `pytest` 或浏览器测试，可在此基础上增量扩展。

### 风险控制

1. 若环境变量覆盖能力不足，优先在配置层补充显式 `LINGSHU_DATA_DIR` 一类入口，而不是在测试里硬编码路径替换。
2. 若子进程启动存在竞争，测试辅助模块需等待健康检查成功后再发业务请求。
3. 若 discovered 样本目录结构与真实外部 CLI 差异过大，测试应采用“最小可解析样本”原则并在 README 中说明边界。

## 风险与待确认项

1. 当前没有现成测试框架报告格式，本轮默认接受 `unittest` 文本输出。
2. 若后续决定接入 CI，可能需要再补充统一脚本入口或退出码约束，但不影响本轮方案成立。
3. 工作流接口校验场景较多，本轮只覆盖关键失败路径，不追求所有非法组合。

## 交接输出

1. 推荐架构方案：`unittest + uvicorn subprocess + urllib.request + 环境变量隔离目录`。
2. 计划阶段需基于以下写集拆分实现：
   - 测试基础设施与隔离配置
   - API 回归用例
   - README 与实现记录
3. 开发阶段若需改动 `config.py`，只能做满足测试隔离所需的最小修改，不得连带重构业务层。
