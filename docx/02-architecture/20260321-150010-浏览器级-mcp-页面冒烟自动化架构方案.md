## 目标

1. 选择浏览器冒烟自动化的最小可执行方案。
2. 把本机缓存浏览器与 CI 标准安装路径同时纳入方案。

## 核心内容

1. 方案对比：
   - 方案 A：引入完整 E2E 框架与复杂交互脚本。
     - 优点：覆盖深。
     - 缺点：超出本轮范围，维护成本高。
   - 方案 B：最小 Playwright 冒烟。
     - 通过 `webServer` 启动真实后端和 Vite 开发服务器。
     - 只断言关键页面标题、核心容器和 MCP 页面基础交互。
     - 把浏览器测试接入 `npm run verify`。
2. 推荐方案：方案 B。
3. 模块边界：
   - `playwright.smoke.config.ts`：定义 Chromium、webServer 和 baseURL。
   - `tests/browser/*.spec.ts`：放最小页面冒烟用例。
   - `package.json`：新增 `test:browser` 并扩展 `verify`。
   - `src/layouts/AppLayout.vue`、`src/pages/McpsPage.vue`：仅补必要的稳定选择器。
   - `.github/workflows/ci.yml`：安装 Playwright Chromium 后运行统一验证。
4. 运行策略：
   - 本机优先复用 `/root/.cache/ms-playwright/.../chrome`。
   - 其它环境若该路径不存在，则退回 Playwright 默认浏览器解析逻辑。
   - 前端使用 `vite dev`，复用 `/api` 代理到 `127.0.0.1:8000`。
5. 断言策略：
   - 页面导航断言：标题和关键列表/区块可见。
   - MCP 页面断言：`Readonly MCP` 与 `Locked MCP` 可见；可写/只读状态切换可见；新建对话框中 `managed` / `external` 选项可见。

## 风险与待确认项

1. 当前浏览器测试仍不覆盖实际保存动作。
2. Workflow 画布相关断言仅限静态元素，不覆盖拖拽。
