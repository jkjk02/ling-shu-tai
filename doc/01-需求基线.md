# 灵枢台需求基线

- 版本：v0.2
- 状态：已确认
- 日期：2026-03-19
- 输入依据：用户任务说明、`/home/alicization/AGENTS.md`、Figma 官方文档《Guide to the Figma MCP server》、用户确认“按默认方案继续”

## 背景

用户希望在当前仓库内新增一套独立项目目录，承载一个可独立开发和运行的 Web 管理平台，用于集中管理多个 CLI 工具及其配置资产，并提供多智能体协作流程的可视化编排能力。

当前实施落点说明：原始范围文档中使用过 `apps/ling-shu-tai` 的描述；结合当前工作区实际落点，后续实现、文档和交付统一以 `/home/lst/ling-shu-tai` 为准。

该平台面向 `codex`、`cludea`、`opencode` 三类 CLI 工具，核心目的是把分散在本地文件系统中的 Skills、模型配置文件（MCPs）以及 Agent 编排能力统一收口到一个管理界面中，降低配置分散、人工维护和多工具切换带来的成本。

根据 Figma 官方关于 MCP server 的说明，Figma MCP 更适合作为设计上下文与开发协同桥梁，由受支持的编码客户端访问 Figma 设计资源。当前需求中明确要求“基于该 MCP 完成 UI 设计”，因此本轮将其纳入 UI 设计约束与交付方式考虑，但不将其实现为平台内的直连功能。

## 目标

1. 提供一个基于 Vue 3 + TypeScript 的 Web 管理界面，统一管理 `codex`、`cludea`、`opencode` 的 Skills 与 MCPs。
2. 提供一个基于 FastAPI 的后端服务，对本地资源进行发现、读取、保存和删除，并通过 RESTful API 向前端提供数据。
3. 支持多智能体 Agent 的定义、配置、技能分配、模型配置关联。
4. 支持多智能体协作流程的可视化编排、保存与加载。
5. 让界面结构、信息层次和组件组织适合后续通过 Figma MCP 工作流继续迭代 UI 设计。
6. 项目以独立目录形式组织，并使用中文项目名“灵枢台”。

## 范围

### 本次纳入范围

1. 新建一个独立项目目录 `apps/ling-shu-tai`。
2. 新建一个前端管理台应用：
   - Vue 3
   - Composition API
   - TypeScript
   - Element Plus
3. 新建一个 Python 3.10+ 后端服务：
   - FastAPI
   - RESTful JSON API
   - 本地文件系统扫描与持久化
4. 实现以下资源的 CRUD：
   - Skill
   - MCP
   - Agent
   - Workflow
5. 实现 UI 页面：
   - Dashboard / 总览
   - Skills 列表与编辑页
   - MCPs 列表与编辑页
   - Agents 列表与编辑页
   - Workflows 编排页
6. 实现后端启动扫描机制：
   - 扫描三类 CLI 工具的标准配置目录
   - 识别已有 Skills 与 MCPs
   - 加载为平台可管理资源
7. 实现前后端联调所需的基础数据模型、错误处理与表单校验。

### 本次不纳入范围

1. 真正执行 CLI 命令并流式监控任务输出的运行时编排系统。
2. 用户权限、登录鉴权、RBAC。
3. 多租户、组织级资源隔离。
4. 与云端数据库或远程对象存储集成。
5. 与真实 Figma 文件进行双向同步或在平台内嵌入 Figma 编辑能力。
6. 复杂的实时协同编辑、多人同时编排工作流。

## 角色与场景

### 角色

1. 平台管理员：
   - 维护各 CLI 工具的 Skills 与 MCPs
   - 调整 Agent 定义与技能授权
   - 管理工作流模板
2. AI 工程师 / 工具链维护者：
   - 新建或修改 Skill 脚本
   - 试验不同模型参数配置
   - 编排多 Agent 协作链路
3. 设计协同人员：
   - 基于 Figma 产出的设计资源继续细化管理台 UI
   - 不直接操作业务配置，但需要界面结构可被 Figma MCP 工作流消费

### 典型场景

1. 管理员进入平台后查看三个 CLI 工具已发现的 Skill 数量、MCP 数量、Agent 数量和 Workflow 数量。
2. 工程师为 `codex` 新建一个 Skill，填写名称、描述、触发命令，并在代码编辑器中编写脚本内容。
3. 工程师为 `opencode` 配置一个新的 MCP，设置 `model_name`、`temperature`、`max_tokens` 等参数。
4. 管理员创建一个 Agent，填写 System Prompt，并从 Skill 库中勾选多个可用 Skill。
5. 管理员在可视化画布中拖入多个 Agent 节点，连接任务流向后保存 Workflow。
6. 平台重启后，系统重新扫描本地配置目录，保留并展示已存在的资源。

## 已确认项

1. 前端技术栈为 Vue 3、Composition API、TypeScript。
2. 后端技术栈为 Python 3.10+、FastAPI。
3. 前后端通信方式为 RESTful API + JSON。
4. 需要覆盖 `codex`、`cludea`、`opencode` 三类 CLI 工具。
5. 需要提供 Skills、MCPs、Agents、Workflows 的可视化管理能力。
6. Workflow 需要支持画布式节点与连线编排，并支持保存和加载。
7. 前端 UI 框架按默认方案使用 `Element Plus`。
8. “监控”本期仅覆盖资源发现状态、数量统计、更新时间等静态监控。
9. Figma MCP 本期仅作为 UI 设计约束，不做平台内实际集成。
10. 存储方式按默认方案采用“文件系统持久化 + 内存索引”，本期不引入数据库。

## 功能需求

### 1. 平台总览

1. 提供 Dashboard 页面，展示：
   - 三个 CLI 工具的资源统计
   - 最近更新时间
   - Agent / Workflow 数量
   - 文件发现状态或最近扫描结果
2. 提供快速入口跳转到 Skills、MCPs、Agents、Workflows。

### 2. CLI 工具资源管理

#### 2.1 Skills 管理

1. 按 CLI 工具维度查看 Skill 列表。
2. 列表视图至少展示：
   - 名称
   - 描述
   - 所属 CLI 工具
   - 更新时间
3. 支持 Skill 的新建、编辑、删除、查看详情。
4. 编辑页至少包含：
   - 名称
   - 描述
   - 所属 CLI 工具
   - 触发命令
   - 脚本内容
5. 脚本内容编辑区使用代码编辑器组件，优先考虑 Monaco Editor。

#### 2.2 MCP 管理

1. 按 CLI 工具维度查看 MCP 列表。
2. 列表视图至少展示：
   - 名称
   - 关联模型
   - 所属 CLI 工具
   - 更新时间
3. 支持 MCP 的新建、编辑、删除、查看详情。
4. 编辑页至少支持以下字段：
   - 名称
   - 所属 CLI 工具
   - `model_name`
   - `temperature`
   - `max_tokens`
   - 其他扩展参数（结构化 JSON 或键值对形式）

### 3. 后端文件发现机制

1. 后端启动时自动扫描 `codex`、`cludea`、`opencode` 的标准配置目录。
2. 支持识别已有 Skills 与 MCP 文件，并导入到平台状态中。
3. 支持平台修改后回写到对应资源文件。
4. 扫描结果应包含成功、失败、缺失目录等状态信息，供前端展示。
5. 若某 CLI 在当前环境未安装或配置目录不存在，平台仍需正常运行，并明确提示该资源未发现。

### 4. Agent 定义与管理

1. 支持 Agent 列表页展示：
   - 名称
   - ID
   - 角色摘要
   - 已分配 Skill 数量
   - 关联 MCP
2. 支持 Agent 的新建、编辑、删除、查看详情。
3. 编辑页至少包含：
   - Agent 名称
   - Agent ID
   - System Prompt 文本区域
   - Skill 多选分配
   - MCP 关联选择
4. Skill 分配应基于模块一中的 Skill 库动态加载。
5. MCP 关联应基于模块一中的 MCP 库动态加载。

### 5. Multi-Agent Workflow 编排

1. 提供基于画布的 Workflow 编辑器。
2. 用户可在画布中：
   - 拖入 Agent 节点
   - 编辑节点基础信息
   - 通过连接线定义任务流转关系
3. 支持 Workflow 的新建、编辑、删除、保存、加载。
4. 支持保存 Workflow 元数据：
   - 名称
   - 描述
   - 节点列表
   - 连线列表
   - 布局信息
5. 工作流画布优先考虑采用 Vue Flow 或同类方案。

### 6. UI/UX 设计要求

1. 信息架构需清晰区分四类资源：
   - Skills
   - MCPs
   - Agents
   - Workflows
2. 列表页、编辑页、详情页应保持统一交互模式。
3. 复杂配置项应具备足够的表单分组、说明文案和错误提示。
4. Workflow 画布需适配桌面优先的管理台交互，并保证基础响应式能力。
5. 本轮 UI 设计应考虑后续通过 Figma MCP 工作流继续迭代，因此组件层级、命名和页面区域应保持清晰，便于与设计稿映射。

## 非功能需求

1. 前端代码使用 Vue 3 Composition API 和 TypeScript。
2. 后端代码使用 Python 3.10+ 与 FastAPI。
3. API 返回 JSON，错误响应结构统一。
4. 前端与后端应可本地独立启动，并提供清晰的开发说明。
5. 后端文件发现逻辑需尽量兼容常见本地目录结构，至少对缺失目录和读取失败提供可诊断提示。
6. 表单交互需要具备基础校验与空值保护。
7. 代码结构需支持未来继续扩展更多 CLI 工具类型。
8. 在不引入数据库的前提下，资源存储策略需清晰可维护。

## 验收标准

1. 启动前端与后端后，用户可以在浏览器中访问一个完整的管理台界面。
2. 用户可以对 Skills、MCPs、Agents、Workflows 执行增删改查。
3. Skill 编辑页提供代码编辑器，能够编辑并保存脚本内容。
4. Agent 编辑页支持从已有 Skill 中多选分配，并关联一个 MCP。
5. Workflow 页支持拖拽 Agent 节点、连接节点、保存并重新加载配置。
6. 后端服务启动时会执行本地资源扫描，并将结果反馈到前端。
7. 当某 CLI 的标准配置目录不存在时，平台不会崩溃，且界面能展示“未发现/未配置”状态。
8. UI 结构完整，适合作为后续通过 Figma MCP 工作流继续深化设计的基础版本。

## 结论

在当前信息下，可以确认灵枢台是一个“前后端分离、以本地文件系统为资源源头、以管理台 CRUD + Workflow 编排为核心”的工具平台需求。

后续架构阶段将默认采用以下方向推进：

1. 前端使用 Vue 3 + TypeScript + Element Plus。
2. 后端使用 FastAPI + Pydantic + 文件系统存储。
3. Workflow 画布使用 Vue Flow。
4. Skill 脚本编辑器使用 Monaco Editor。
