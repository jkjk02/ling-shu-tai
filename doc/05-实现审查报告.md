- 日期：2026-03-19
- 输入依据：[01-需求基线.md](/home/lst/ling-shu-tai/doc/01-需求基线.md)、[02-架构方案.md](/home/lst/ling-shu-tai/doc/02-架构方案.md)、[04-实现记录.md](/home/lst/ling-shu-tai/doc/04-实现记录.md)、当前代码实现

## 总体评估

当前评级：中。

`Skills` 纵向切片已经从“只有列表占位”推进到“具备真实 CRUD 闭环”的状态，方向与上一轮审查建议一致，且前后端职责边界比首轮骨架更清晰。

但从整个平台视角看，`Workflow`、外部资源发现回写、`MCPs/Agents` 的编辑闭环和自动化测试仍然不足，因此当前版本仍属于“局部能力成形、整体验收尚未达标”的阶段。

## 核心问题与风险

### 1. 高：Workflow 核心能力仍未落地，平台差异化价值尚未形成

- 影响：
  - 直接影响需求基线中的画布编排、拖拽连线、保存和加载验收标准。
  - 若继续延后，平台会长期停留在“资源管理页集合”，而不是“多智能体协作平台”。
- 证据：
  - [WorkflowsPage.vue](/home/lst/ling-shu-tai/src/pages/WorkflowsPage.vue) 仍以占位画布和静态节点渲染为主。
  - [resources.ts](/home/lst/ling-shu-tai/src/api/resources.ts) 当前仍未补齐 Workflow 的写操作接入。
- 建议：
  - 将 Workflow 作为下一优先级最高的工程项，至少补齐节点拖拽、连线、保存、加载的最小闭环。

### 2. 高：外部资源发现与回写仍停留在目录扫描和项目内 JSON 仓储层面

- 影响：
  - 当前平台只能稳定管理项目内示例数据，尚未真正管理用户本地 CLI 的既有资源。
  - 这会使需求基线中的“自动发现、导入、回写”承诺继续落空。
- 证据：
  - [discovery.py](/home/lst/ling-shu-tai/backend/app/services/discovery.py) 仍以路径检测与 JSON 数量统计为主。
  - [resource_service.py](/home/lst/ling-shu-tai/backend/app/services/resource_service.py) 仍以项目内聚合 JSON 文件作为主要持久化目标。
- 建议：
  - 先从 `Skills` 与 `MCPs` 入手，明确支持格式、只读策略和回写路径，再逐步扩展。

### 3. 中：`MCPs`、`Agents`、`Workflows` 页面仍未复制 `Skills` 的真实 CRUD 模式

- 影响：
  - 当前只有 `Skills` 形成了相对完整的列表、详情、编辑、删除模式，其余资源仍然不满足需求基线中的 CRUD 口径。
  - 如果各页后续各自独立演化，容易导致交互风格、字段约束和错误处理不一致。
- 证据：
  - [McpsPage.vue](/home/lst/ling-shu-tai/src/pages/McpsPage.vue)、[AgentsPage.vue](/home/lst/ling-shu-tai/src/pages/AgentsPage.vue)、[WorkflowsPage.vue](/home/lst/ling-shu-tai/src/pages/WorkflowsPage.vue) 仍以展示和占位为主。
- 建议：
  - 复用 `Skills` 页的页面模式、表单结构和错误提示策略，优先推进 `MCPs`，再推进 `Agents`。

### 4. 中：`Skills` 脚本编辑区当前仍是多行文本域，而非需求基线中优先约定的 Monaco Editor

- 影响：
  - 当前已经可以编辑与保存脚本内容，但缺少语法高亮、行号、代码折叠等编辑体验。
  - 若后续脚本内容变长，文本域会明显影响维护效率。
- 证据：
  - [SkillsPage.vue](/home/lst/ling-shu-tai/src/pages/SkillsPage.vue) 使用 `el-input type="textarea"` 作为脚本编辑区。
- 建议：
  - 在 `Skills CRUD` 稳定后，再补接 Monaco，避免在核心闭环未稳时过早引入较重依赖。

### 5. 中：测试与回归防线仍偏弱，当前主要依赖构建和进程内冒烟

- 影响：
  - 已补上的 `Skills` 闭环如果没有自动化回归，很容易在后续复制到其他资源页时被破坏。
  - 当前仍缺少 HTTP 层和浏览器层的黑盒验证。
- 证据：
  - [package.json](/home/lst/ling-shu-tai/package.json) 仍无测试脚本。
  - 本轮验证主要是 `npm run build` 与进程内路由函数级冒烟。
- 建议：
  - 优先为 `Skills` 的后端路由和前端 DTO/页面交互补一条最小回归链路。

## 已解决项

1. 上一轮审查中关于“前端资源页只有占位，没有真实 CRUD”的高优先级问题，至少在 `Skills` 资源上已经得到解决。
2. 上一轮审查中关于“`resources.ts` 只有列表接口，缺少写操作接入”的问题，已经在 `Skills` 维度完成修复。
3. 后端创建静默覆盖与只读资源被误写的风险，已经通过冲突校验和只读保护显式收紧。

## 亮点与最佳实践

1. `Skills` 页采用“列表 + 详情 + 对话框表单”的模式，已形成后续可复制到其他资源页的模板。
2. 后端在不引入额外复杂度的前提下，把重复创建冲突和只读资源保护收敛到了服务层，约束边界更清晰。
3. 页面在 mock 回退状态下显式禁用写操作，避免把示例数据演示误判为真实保存成功。

## 残余风险

1. 当前仍未完成真实 HTTP 黑盒验证，结构化错误响应虽然已在代码中补强，但尚未通过浏览器请求链路验实。
2. 前端构建仍提示公共 chunk 偏大，后续如果引入 Monaco 或 Vue Flow，需要继续关注包体积。
3. `Skills` 的 ID 仍由后端按名称 slug 生成，当前策略足够支撑本轮，但跨资源统一 ID 策略后续仍应收敛。

## 结论

本轮实现质量较上一轮明显提升，`Skills` 资源已经具备继续联调和复制模式的基础。

但从需求基线的最终验收角度看，当前只是把最高优先级问题中的一项切实推进了，还不足以宣称整体完成。下一轮应优先在 `MCPs` 或 `Workflow` 上延续同样的纵向闭环策略，而不是重新回到横向铺骨架。
