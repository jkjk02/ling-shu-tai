# 灵枢台实施计划

- 版本：v0.1
- 状态：已完成
- 日期：2026-03-19
- 输入依据：[01-需求基线.md](/home/lst/ling-shu-tai/doc/01-需求基线.md)、[02-架构方案.md](/home/lst/ling-shu-tai/doc/02-架构方案.md)

## 背景

本计划用于指导灵枢台在独立项目目录内完成前后端实现、验证和交付。

## 核心内容

### 阶段 1：项目骨架与基础设施

1. 创建项目根目录 `apps/ling-shu-tai/`。
2. 搭建前端 Vite + Vue 3 + TypeScript 项目骨架。
3. 搭建 FastAPI 后端骨架。
4. 建立 `doc/`、`backend/data/` 和前端功能目录。

交付物：

1. 前端可启动空壳管理台。
2. 后端可启动健康检查接口。

### 阶段 2：统一资源模型与 API

1. 定义 Skill、MCP、Agent、Workflow 的共享模型与 DTO。
2. 实现 REST API：
   - `/api/dashboard`
   - `/api/discovery`
   - `/api/skills`
   - `/api/mcps`
   - `/api/agents`
   - `/api/workflows`
3. 建立统一错误响应结构。

交付物：

1. 后端具备完整 CRUD 能力。
2. 前端可通过 API 获取真实数据。

### 阶段 3：文件发现与持久化

1. 实现 CLI 路径候选扫描。
2. 实现文件格式解析与错误记录。
3. 实现外部文件回写与 managed 目录兜底保存。
4. 增加 Figma MCP 预置示例配置。

交付物：

1. 启动后可看到发现状态。
2. 至少能管理项目内保存的资源和发现到的外部资源。

### 阶段 4：前端功能页

1. Dashboard 页面
2. Skills 页面
3. MCPs 页面
4. Agents 页面
5. Workflow 画布页面

组件拆分原则：

1. 路由页只做编排。
2. 列表、表单、详情、画布拆成独立功能组件。
3. API 请求与状态放入 composables。

### 阶段 5：联调、审查与测试

1. 完成前后端联调。
2. 做基础手工验证与关键自动化测试。
3. 输出实现记录、审查报告、测试报告和交付摘要。

## 实施拆分建议

### 前端

1. 先做 App Shell、导航和 Dashboard。
2. 再做 Skills / MCPs 的表格 + 对话框 CRUD。
3. 然后做 Agent 绑定 Skill/MCP 的复合表单。
4. 最后实现 Vue Flow 画布和 Workflow 序列化。

### 后端

1. 先完成 schemas 和健康检查。
2. 再完成 storage/discovery 两个核心服务。
3. 接着实现 CRUD 路由。
4. 最后补充扫描状态与示例种子数据。

## 风险与应对

1. 依赖安装风险：
   - 风险：当前仓库未预装 Element Plus、Vue Flow、Monaco、FastAPI。
   - 应对：最小化新增依赖，并优先验证安装与类型检查。
2. 真实外部目录格式差异风险：
   - 风险：不同 CLI 工具的文件结构可能不统一。
   - 应对：先做候选路径与最佳努力解析，确保平台自身 managed 目录始终可用。
3. 画布复杂度风险：
   - 风险：Workflow 交互容易膨胀。
   - 应对：本期只做节点拖拽、连线、保存与加载，不做复杂执行语义。

## 结论

灵枢台适合按“骨架 -> 模型/API -> 文件发现 -> 前端功能 -> 联调测试”的顺序推进。该顺序可以把风险前置，并确保每一步都有可验证的阶段结果。
