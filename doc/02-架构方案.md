# 灵枢台架构方案

- 版本：v0.1
- 状态：已完成
- 日期：2026-03-19
- 输入依据：[01-需求基线.md](/home/lst/ling-shu-tai/doc/01-需求基线.md)

## 背景

灵枢台需要在一个独立项目目录内同时承载前端管理台、FastAPI 后端、基于本地文件系统的资源发现与持久化能力，并为 Skills、MCPs、Agents、Workflows 四类资源提供统一的管理体验。

## 方案候选

### 方案 A：独立项目目录下的前后端一体仓结构

- 目录：`apps/ling-shu-tai/`
- 前端：Vue 3 + Vite + Element Plus
- 后端：FastAPI
- 存储：本地文件系统 + 内存索引
- 工作流画布：Vue Flow

优点：

1. 满足“独立项目目录”要求，前后端和文档聚合在同一根目录。
2. 目录边界清晰，便于交付、维护和后续迁移。
3. 对当前 monorepo 干扰最小，不需要引入数据库或复杂基础设施。

缺点：

1. Python 后端不完全复用现有 Node 服务体系。
2. 需要额外维护一套 Python 依赖与启动说明。

### 方案 B：前端放 `apps/`，后端拆到 `services/`

- 前端：`apps/<web-app>`
- 后端：`services/<fastapi-service>`
- 存储：共享文件目录

优点：

1. 更贴近仓库现有 apps/services 分层方式。
2. 后端未来可独立部署或扩展成独立服务。

缺点：

1. 不满足用户强调的“重新创建项目文件夹里面”这一组织偏好。
2. 文档、前端、后端分散，阶段文档和交付边界不够聚合。
3. 当前项目规模下会增加协调成本。

## 方案对比

1. 从目录聚合度看，方案 A 更适合本轮独立交付。
2. 从实现成本看，方案 A 能以更少改动完成需求。
3. 从长期演化看，方案 B 更利于服务拆分，但超出本轮需求。

## 推荐方案

推荐采用方案 A，在 [ling-shu-tai](/home/lst/ling-shu-tai) 下建立独立项目目录。

推荐原因：

1. 与已确认需求和新增组织要求完全一致。
2. 便于把文档、代码、数据目录统一落盘。
3. 有利于先完成“可用且可维护”的基础平台，再按需要拆分服务。

## 设计原则与权衡

1. 低成本优先：不引入数据库、不引入复杂鉴权、不引入运行时任务系统。
2. 可演化优先：统一资源模型和路径适配器，为未来新增 CLI 工具和数据库留接口。
3. 独立目录优先：所有与灵枢台直接相关的前后端与文档聚合在同一个项目根下。
4. 文件系统优先：以本地文件为真源，后端以内存索引提升读取效率。

## 总体架构设计

```text
浏览器
  -> Vue 3 管理台
     -> REST API
        -> FastAPI 应用
           -> 发现服务 DiscoveryService
           -> 持久化服务 StorageService
           -> 资源仓储 Repositories
              -> 外部 CLI 配置目录
              -> 项目内 managed 数据目录
```

### 前后端边界

1. 前端只负责编排页面、表单、列表、画布和 API 调用。
2. 后端负责资源发现、文件解析、统一数据模型映射、持久化与错误归一。
3. 前后端通过 RESTful JSON 通信，不共享运行时状态。

## 核心组件

### 1. 前端管理台

建议目录：

```text
apps/ling-shu-tai/
  src/
    api/
    components/
      dashboard/
      skills/
      mcps/
      agents/
      workflows/
      shared/
    composables/
    layouts/
    pages/
    router/
    styles/
```

组件边界：

1. `pages/` 只做路由级编排。
2. `components/<feature>/` 承载功能型 UI。
3. `composables/` 负责资源获取、提交、筛选和画布状态。
4. `api/` 统一封装请求和 DTO 类型。

### 2. 后端服务

建议目录：

```text
apps/ling-shu-tai/backend/
  app/
    routers/
    schemas/
    services/
    repositories/
    utils/
    config.py
    main.py
  data/
    managed/
      skills/
      mcps/
    agents/
    workflows/
```

职责划分：

1. `routers/` 暴露 REST API。
2. `schemas/` 定义 Pydantic 输入输出模型。
3. `services/discovery` 扫描本地目录并归一化外部资源。
4. `services/storage` 负责项目内资源读写、外部资源回写与冲突处理。
5. `repositories/` 聚合文件读写与资源索引。

### 3. 文件存储目录

统一采用“外部发现 + 项目托管”双来源模型：

1. 外部发现来源：
   - CLI 工具自己的标准配置目录
2. 项目托管来源：
   - `backend/data/managed/skills/<tool>/`
   - `backend/data/managed/mcps/<tool>/`
   - `backend/data/agents/`
   - `backend/data/workflows/`

## 文件发现机制

### 候选标准路径

使用“候选路径列表 + 环境变量覆盖 + 不存在即降级”的策略。

默认候选根目录：

1. `codex`
   - `~/.codex`
   - `~/.config/codex`
2. `cludea`
   - `~/.cludea`
   - `~/.config/cludea`
   - `~/.claude`
   - `~/.config/claude`
3. `opencode`
   - `~/.opencode`
   - `~/.config/opencode`

资源子目录候选：

1. Skills：
   - `skills/`
   - `commands/`
   - `prompts/skills/`
2. MCPs：
   - `mcps/`
   - `mcp/`
   - `models/`
   - `config/mcps/`

### 发现策略

1. 优先读取环境变量显式覆盖路径。
2. 依次遍历候选根目录和资源子目录。
3. 对支持的文件类型做最佳努力解析：
   - Skill：`.md`、`.txt`、`.json`、`.yaml`、`.yml`、`.toml`、`.sh`
   - MCP：`.json`、`.yaml`、`.yml`、`.toml`
4. 解析失败时保留错误信息，但不终止整次扫描。
5. 若全部路径不存在，则返回 `missing` 状态。

### 降级策略

1. 外部目录不存在：
   - 平台继续启动
   - 仪表盘展示“未发现”
2. 文件存在但解析失败：
   - 保留原文件路径与错误信息
   - 前端提示“发现失败”
3. 外部文件不可写：
   - 允许查看
   - 禁止直接覆盖
   - 改为另存到 managed 目录

## 技术选型与理由

### 前端 UI：Element Plus

推荐理由：

1. 管理台组件完整，表格、表单、抽屉、对话框、标签、空状态都现成。
2. 与 Vue 3 + TypeScript 集成成熟，降低基础 UI 组装成本。
3. 更适合信息密度较高的后台界面。

备选方案：Naive UI

优点：

1. TS 类型体验较好。
2. 主题扩展灵活。

缺点：

1. 对本轮后台场景的现成范式沉淀不如 Element Plus。
2. 团队复用管理台模板和资料时，Element Plus 更通用。

### 工作流画布：Vue Flow

推荐理由：

1. 与 Vue 3 生态匹配直接。
2. 节点、边、视口、拖拽和保存序列化能力成熟。
3. 能快速完成“可编辑工作流画布”的核心需求。

### 代码编辑器：Monaco Editor

推荐理由：

1. 适合 Skill 脚本和 JSON 配置编辑。
2. 具备语法高亮和较好的编辑体验。
3. 与管理台复杂配置场景匹配。

### Figma MCP 示例

根据 Figma 官方文档，远程服务地址为 `https://mcp.figma.com/mcp`，桌面服务地址通常为 `http://127.0.0.1:3845/mcp`。本项目可将其作为 MCP 管理模块中的预置示例配置，帮助用户管理与 UI 设计协同有关的 MCP 记录。

## 数据架构设计

### 持久化方案

不引入数据库，采用“文件系统持久化 + 启动扫描重建索引 + 请求内内存缓存”：

1. Skills / MCPs
   - 外部资源优先回写原文件
   - 不可回写时保存到 managed 目录
2. Agents / Workflows
   - 统一保存为项目内 JSON 文件
3. 启动时扫描并构建资源索引
4. 写操作后刷新对应资源索引

### 为什么不引入数据库

1. 当前资源规模可控，数据主要来自本地配置文件。
2. 文件系统就是业务真源，引入数据库会造成双写与同步复杂度。
3. 本期目标是低成本交付可用平台，而不是做中心化 SaaS。

## 非功能需求保障

1. 可维护性：
   - 统一资源模型
   - 前后端明确分层
2. 可扩展性：
   - 路径扫描器按 CLI 适配器组织
   - 新增工具只需补充候选路径与解析器
3. 可用性：
   - 任一工具目录缺失不影响系统启动
4. 安全性：
   - 当前仅做本地管理台，不暴露远程写接口权限模型
   - 路径写入限制在候选目录与项目 managed 目录内

## 风险与权衡

1. `cludea` 命名可能与真实 CLI 不一致：
   - 处理：保留配置化候选路径，并兼容 `claude` 别名路径。
2. 外部文件格式不统一：
   - 处理：采用最佳努力解析和错误可视化，不强行统一外部文件规范。
3. 无数据库下的并发写入能力有限：
   - 处理：当前按单机本地工具平台定位接受该限制。
4. 前端依赖数量会增加：
   - 处理：只引入 Element Plus、Vue Flow、Monaco 等最小必需集。

## 结论

灵枢台当前最合适的架构是：以 [apps/ling-shu-tai](/home/alicization/apps/ling-shu-tai) 为独立项目根目录，采用 Vue 3 前端 + FastAPI 后端 + 文件系统持久化 + 内存索引 + 适配器式路径发现机制。该方案在当前约束下成本最低、实现最快、后续也具备继续演化为更完整工具平台的空间。
